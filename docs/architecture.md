# 系统架构

这个项目是一个 Codex-native 的因果推断与政策评估工作台。它不把 Codex 当作聊天机器人，而是把 Codex 作为项目内的研究代理来运行。

## 层次

1. **项目宪法**：`AGENTS.md` 定义角色、方法路由、质量门禁和报告格式。
2. **技能层**：`.codex/skills/*/SKILL.md` 把常用研究任务封装成可调用 workflow。
3. **专家代理层**：`.codex/agents/*.toml` 定义政策来源搜集、政策文件处理、事件侦察、识别审查、数据工程、估计、稳健性和政策翻译等角色。
4. **安全与上下文层**：`.codex/hooks.json` 与 `.codex/hooks/*.py` 为危险命令、研究上下文和失败命令提供拦截或提示。
5. **知识库**：`knowledge/*.md` 记录方法知识、方法矩阵、政策发现手册、政策文件处理协议、事件库和数据源。
6. **可执行工具层**：`scripts/*.py` 与 `src/causal_policy_system/*` 提供政策文件抽取、方法推荐、design card 验证、demo 数据、基线估计和报告渲染。
7. **研究实例层**：`studies/` 存放每个政策评估项目的卡片、数据、脚本、输出和报告。

## 典型流转

政策问题 → 政策来源搜集 → 政策文件抽取/核验 → intake → method routing → design card → 数据审计 → 估计 → 诊断/稳健性 → 反方审查 → 报告/政策 memo → 复现包。

## 为什么使用规则路由 + LLM 审查

方法推荐器是透明的规则系统，适合作为第一反应；Codex agents 再根据制度背景、数据可得性和文献证据覆盖规则系统无法判断的部分。这种设计避免把“模型觉得像 DID”误当成完整的识别论证。
